# Общий план разработки

### Этап 1. Пре-продакшн (Планирование и Математика)

В отличие от сюжетных игр, здесь мы не пишем сценарий, а строим экономику. Без этого этапа начинать кодить кликер бесполезно.

**1. Концепт и "Крючок" (The Hook)**

* Определить сеттинг (печеньки, хомяки, космос, завод).
* Придумать главную фишку: почему игрок не закроет игру через 5 минут? (Пример: "Кликер смешанный с RPG-боями" или "Кликер, где ты строишь реальный город").

**2. Математическое моделирование (Excel/Google Sheets)**

* Это твой "Геймдизайн-документ". Ты создаешь таблицу баланса.
* Рассчитываешь прогрессию на первые 3 дня игры: сколько кликов нужно для 1-го здания? Сколько секунд ждать 2-го?
* **Результат:** Графики, показывающие, что игроку всегда есть что купить ближайшие 5 минут.

**3. Прототипирование UI (Интерфейса)**

* Кликер — это на 90% интерфейс.
* Рисуем "вайрфреймы" (черновые схемы): где кнопка, где счетчик валюты, где магазин. Важно понять, не перегружен ли экран, если механик много.

**4. Выбор Технического Стека**

* Решение проблемы больших чисел (BigInt).
* Выбор базы данных (CloudStorage для Telegram или свой сервер).

---

### Этап 2. Продакшн (Разработка слоями)

Здесь мы применяем **итеративный подход**. Мы не делаем все механики сразу, а наслаиваем их друг на друга.

**Слой 1: Core Loop (Базовый цикл)**

* **Код:** Реализация клика, начисления валюты, покупки первого здания.
* **Арт:** Отрисовка основной кнопки и иконки валюты.
* **Задача:** Сделать так, чтобы нажимать было приятно (визуальные эффекты, звуки).

**Слой 2: Idle-составляющая (Автоматизация)**

* **Код:** Реализация пассивного дохода.
* **Система сохранений:** Критически важный момент. Запись прогресса каждые 10-30 секунд и расчет "оффлайн-дохода" (сколько заработано, пока игра была закрыта).

**Слой 3: Мета-игра (Глубина)**

* Внедрение тех самых "интересных механик", о которых ты говорил.
* **RPG-элементы:** Инвентарь, герои, прокачка.
* **Престиж (Rebirth):** Механика сброса прогресса ради бонусов.
* **Социалка:** Лидерборды, кланы, реферальная система (особенно важно для Telegram Mini App).

**Слой 4: Полиш (Polishing)**

* Добавление "сочности" (Juice): анимации вылетающих цифр, красивые прогресс-бары, всплывающие окна.
* Обучение (Tutorial): Первая сессия игрока должна вести его за руку.

---

### Этап 3. Тестирование и Балансировка (Soft Launch)

Для кликера этот этап важнее, чем для других жанров.

**1. Технический тест**

* Проверка сохранений: что будет, если интернет пропадет во время покупки?
* Нагрузочный тест: выдержит ли игра триллионы вычислений?

**2. Плейтесты баланса**

* Ты даешь игру тестерам и смотришь в аналитику.
* **Главный вопрос:** На какой минуте игроки закрывают игру? Если на 10-й минуте, значит, там "стена" (слишком дорогой апгрейд), нужно править Excel-таблицу.

---

### Этап 4. Релиз и Пост-продакшн (LiveOps)

Кликер — это игра-сервис. Она живет, пока ты её обновляешь.

**1. Релиз (Launch)**

* Публикация в сторе (или запуск бота). Маркетинговый "взрыв".

**2. LiveOps (Оперирование)**

* **События (Events):** Временные ивенты (например, "Турнир выходного дня" или "Хэллоуин"), чтобы игрокам было не скучно.
* **Контент-пачи:** Добавление новых "тира" (tier) зданий, новых героев, новых веток прокачки. В кликере контент заканчивается быстро, его нужно постоянно доливать.

**3. Аналитика и Монетизация**

* Тонкая настройка рекламы и покупок. Где игрок готов заплатить, чтобы ускорить прогресс?

# Руководство по оформлению кода

> НО не надо писать подробные докстринги, подробные только там, где это необходимо!

Это документ, который призван обеспечить единый подход к написанию кода в нашем проекте. Это упростит чтение и понимание кода всеми участниками команды, что положительно повляет на будущий редактирование этого же кода.

Ниже изложены основные правила и рекомендации, которые нужно соблюдать при написании, рефакторинге и обзоре (code review) кода.

---

## Обучающее видео

Перед началом обязательно посмотрите и постарайтесь запомнить ключевые идеи из данного видео:
[Плохой vs. хороший код / wtf is clean code](https://www.youtube.com/watch?v=A1s-2AsBIkY)

В этом видео приводятся общие принципы «чистого кода», которые помогут вам выработать лучшие практики программирования вне зависимости от конкретного проекта.

## 1. Общие требования к структуре проекта и файлов

### 1.1 Структура директорий
- Основной код проекта должен находиться в папке `src/`.  
- Каждую логическую часть выносить в отдельный модуль или пакет (подпапку):
  - Пример:  
    - `chat_gpt_agent.py` — класс ChatGPTAgent.  
    - `message_context.py` — класс MessageContext.  
- Конфигурационные файлы (например, `config.json`) и документацию (`README.md`) хранить в корне проекта.

### 1.2 Именование файлов
- Python-модули (файлы .py) именовать в стиле `snake_case` (например, `data_processor.py`).  
- Все созданные файлы должны иметь говорящие названия, отражающие их назначение (например, `config.json`, `logging.yaml`).

### 1.3 Содержание файла
- Стремитесь к принципу «один файл = одна логическая сущность» либо группа тесно связанных объектов.  
- Если логика в файле становится слишком большой и разнородной, её следует вынести в дополнительные модули.

---

## 2. Общий стиль кода и PEP 8 (не обязательно)

### 2.1 Форматирование
- Использовать 4 пробела для отступов (не табуляцию).  
- Стараться, чтобы длина строки не превышала 100–120 символов. При необходимости использовать переносы.  
- Пустые строки использовать для логического разделения кода и повышения его читабельности.

Пример:
```python
def example_function():
    long_string = (
        "Это пример переноса длинной строки, "
        "чтобы не выходить за рекомендованный лимит."
    )
    print(long_string)
```

### 2.2 Импорты
- Группировать по порядку:  
  1. Стандартные библиотеки Python.  
  2. Сторонние библиотеки.  
  3. Модули текущего проекта.  
- Каждый блок отделять пустой строкой.  
- Проверять, чтобы не оставалось неиспользуемых импортов.

Пример:
```python
# Стандартные библиотеки
import os
import sys

# Сторонние библиотеки
import requests

# Модули текущего проекта
from src import chat_gpt_agent
```

---

## 3. Именование сущностей (обязательно)

### 3.1 Классы
- Использовать стиль CamelCase:  
  ```python
  class UserDatabaseManager:
      ...
  ```

### 3.2 Функции и методы
- Придерживаться `snake_case` для имён методов и функций:  
  ```python
  def process_message():
      ...
  ```
- Приватные (служебные) методы начинаются с одного или двух символов подчёркивания:
  ```python
  def _helper_function():
      ...
  ```
  или
  ```python
  def __private_method():
      ...
  ```

### 3.3 Переменные и константы
- Локальные переменные в `snake_case` (например, `user_name`, `max_token_count`).  
- Приватные переменные класса могут начинаться с подчёркивания (например, `_cache`).  
- Константы в `UPPER_CASE` (например, `MAX_USERS`).

### 3.4 Исключения и краткие описания
- Избегать неочевидных сокращений (например, `usrN` вместо `user_name`).  
- Однобуквенные имена допустимы в коротких блоках кода для счётчиков циклов (`i`, `j`) или математических выражений (`x`, `y`).

## 4. Документирование (очень желательно)

### 4.1 Использование docstrings
- Каждый класс и публичный метод должен иметь docstring с описанием:  
  - Назначения.  
  - Параметров с указанием типа.  
  - Возвращаемого значения (`:return:`).  
  - Исключений (`:raises:`), если таковые выбрасываются. (не обязательно)

Пример:
```python
def send_message(message: str, user_id: int) -> bool:
    """
    Отправка сообщения пользователю.

    :param message: Текст сообщения
    :param user_id: Идентификатор пользователя
    :return: True, если отправка прошла успешно, иначе False
    :raises ValueError: если message пустое или user_id некорректен
    """
    ...
```

Причём важно то, что мы докстринги формата:
```python
def find_meta_for_phase(self, phase: str) -> Optional[MessageMetaData]:  
    """  
    Ищет MessageMetaData, соответствующий указанной фазе.  
    Args:        phase: Фаза/этап алгоритма для поиска в метаданных.  
    Returns:        Объект MessageMetaData или None, если не найден.    
    """
```

Считаем неверными, а верными докстринги следующего формата:
```python
def response_from_chat_GPT(self, user_message: str, images: list = None,  
                           response_format: Optional[Type[BaseModel]] = None) -> str:  
    """  
    Добавляет сообщение пользователя, получает ответ от чата GPT и добавляет его в контекст.  
    :param user_message: Сообщение пользователя для добавления в контекст и отправки в API.    :param images: Список изображений (если есть).    :param response_format: Pydantic модель для парсинга ответа (если требуется).    :return: Ответ ассистента.    
    """
```

или

```python
def is_ancestor(anc, desc):  
    """Проверка: является ли anc префиксом desc. Пустой массив (Исходная) тоже считается предком любого."""
```

для случаев, когда функция не является важной и описание переменных кажется излишним.

### 4.2 Комментарии в коде
- Поясняйте логику, если она неочевидна.  
- Важно описывать «зачем» решено так поступить, а не «что» делает каждая строчка.

---

## 5. Обработка ошибок (желательно)

### 5.1 Исключения
- При возникновении исключений лучше выбрасывать осмысленный класс (например, `ValueError`, `RuntimeError`) и сопровождать коротким пояснением:
  ```python
  if not user_input:
      raise ValueError("Параметр user_input не может быть пустым.")
  ```

### 5.2 Логирование
- Использовать модуль `logging` вместо `print()` для записи ошибок, предупреждений и важной отладочной информации.  
- Устанавливать подходящий уровень (DEBUG, INFO, WARNING, ERROR, CRITICAL) в зависимости от контекста.

## 6. Рекомендуемая структура проекта

Ниже пример общей структуры проекта:
```
project_root/
├── config.json
├── src/
│   ├── __init__.py
│   ├── message_context.py
│   ├── chat_gpt_agent.py
│   └── utils/
│       └── image_processor.py
└── tests/
    └── test_message_context.py
```
- В `src/` лежат основные модули. При необходимости можно создавать подпапки (например, `utils/`).  
- В `tests/` размещать тесты (в том числе unittest/pytest).  
- Файл `config.json` (и другие конфигурационные файлы) хранить в корне, чтобы не путать с кодом.  
- Любые секретные данные должны быть в `.gitignore` (не коммитить API-ключи, пароли и т.п.).

## Обязательные файлы

### config.json

В каждом проекте обязательно должен быть конфигурационный файл config.json с описанием констант программы.


Пример config.json:
```json
{
    "api_key": "",
    "organization": "",
    "model_name": "gpt-4o",
    "temperature": 0.0,
    "max_response_tokens": 4095,
    "max_total_tokens": 30000,

```

### requirements.txt

`requirements.txt` — это файл, в котором перечислены все зависимости (библиотеки и их версии), необходимые для работы проекта на Python. 

Каждый отдельный проект должен быть запущен в своём виртуальном окружении.

При первом открытии скаченного проекта необходимо установить все нужные пакеты с помощью команды `pip install -r requirements.txt`.

requirements.txt создаётся одной командой в терминале: `pip freeze -> requirements.txt`

### ⚙️ LOGIC: THE "LIVING ROOM" SYSTEM

#### 1. Механика Времени (Time System)

Игра проверяет время на телефоне Тани (`new Date().getHours()`) при загрузке.

* **Logic:**
* `06:00 - 18:00` = **Day Mode** (Окно: Светлое небо, CSS: яркость 100%).
* `18:00 - 22:00` = **Evening Mode** (Окно: Розовое/Оранжевое, CSS: теплый фильтр).
* `22:00 - 06:00` = **Night Mode** (Окно: Темно-синее, CSS: синий оверлей + включена гирлянда).


* **The "Control" Switch (Важно для Тани):**
* В комнате стоит Лампа/Торшер.
* *Действие:* Клик по лампе принудительно включает/выключает свет (переключает CSS-класс), игнорируя реальное время.
* *Смысл:* "Я управляю уютом в этом мире".



#### 2. Поведение NPC (Mansur State Machine)

Мансур в игре — это **State Machine** (Конечный автомат). Он никогда не уходит из комнаты ("Я всегда рядом"), но меняет позы.

**Список состояний (States):**

1. **CODING (Base, 60%):** Сидит спиной/боком, в наушниках, быстро печатает. Над головой летят символы кода. (Генерирует валюту пассивно).
2. **RELAX (20%):** Откинулся в кресле, пьет чай/кофе, смотрит в телефон (на "наш чат").
3. **SLEEP (10%):** Спит на диванчике или прямо лицом в клавиатуру (мило).
4. **NOTICE_YOU (10%):** Поворачивается к экрану, машет рукой или показывает сердечко. (Триггерится, если Таня активно кликает).

**Алгоритм смены:**

* При входе в игру выбирается состояние (рандом).
* Каждые 5-10 минут состояние меняется.
* *Исключение:* Если наступила Ночь (реальная), шанс состояния `SLEEP` повышается до 50%, а `CODING` падает.

#### 3. Техническая реализация (Для LLM)

* **Спрайты:** Нам нужно сгенерировать Мансура в 4 позах (один ракурс, изометрия).
* **Код:** Используем `React.useEffect` для проверки времени и `setTimeout` для смены поз. Никакого бэкенда.

#### 2. The Logic: `SaveManager` Class (ТЗ для LLM)

```typescript
// Концепт для генерации кода

const SAVE_KEY = "safe_space_save";

class SaveManager {
  
  // АВТО-СОХРАНЕНИЕ
  // Запускается каждые 10 секунд и при закрытии вкладки (window.onbeforeunload)
  static save(state: GameState) {
    const dataToSave = {
      version: "1.0",
      timestamp: Date.now(), // Важно для Offline расчета
      wallet: state.wallet,
      inventory: state.inventory,
      upgrades: state.upgrades,
      persistent: state.persistent, // Мята и Галерея
      settings: state.settings
    };
    
    localStorage.setItem(SAVE_KEY, JSON.stringify(dataToSave));
  }

  // ЗАГРУЗКА И OFFLINE РАСЧЕТ
  static load(): LoadResult {
    const raw = localStorage.getItem(SAVE_KEY);
    if (!raw) return null; // Новая игра

    const data = JSON.parse(raw);
    
    // MIGRATION CHECK (На будущее)
    // Если data.version < current.version, запускаем скрипт миграции
    
    return data;
  }

  // РАСЧЕТ ОФФЛАЙН ДОХОДА
  static calculateOfflineProfit(lastTime: number, incomePerSec: number): number {
    const now = Date.now();
    const secondsGone = (now - lastTime) / 1000;
    
    // Ограничение: Максимум 24 часа дохода (чтобы не переполнило переменные)
    const effectiveSeconds = Math.min(secondsGone, 86400); 
    
    return Math.floor(effectiveSeconds * incomePerSec);
  }
}

```

### 👆 1.4.3 INTERACTION DESIGN & HAPTICS

#### A. HAPTIC FEEDBACK (Вибрация)

Так как мы в Telegram WebApp, мы используем нативные возможности телефона. Это киллер-фича для погружения.

| Событие | Тип вибрации (Telegram API) | Описание ощущения |
| --- | --- | --- |
| **Клик (Фарм)** | `HapticFeedback.impactOccurred('light')` | Мягкий, как щелчок мыши. Не раздражает при спаме. |
| **Покупка (Shop)** | `HapticFeedback.notificationOccurred('success')` | Приятная "двойная" волна. Чувство достижения. |
| **Открытие меню** | `HapticFeedback.selectionChanged()` | Еле заметный "тик", как при прокрутке барабана. |
| **Ошибка (Мало денег)** | `HapticFeedback.notificationOccurred('error')` | Резкое "Нет". |

#### B. THE "JUICY CLICK" (Визуальный отклик)

Когда Таня касается экрана, интерфейс должен реагировать мгновенно.

1. **Сжатие (Squish):**
* При `mousedown` / `touchstart` вся комната (или курсор) уменьшается на 2% (`scale(0.98)`).
* При `mouseup` — пружинит обратно (`scale(1.02)` -> `scale(1.00)`).


2. **Частицы (Particles):**
* Из точки клика вылетает `+1` и иконка ❤️.
* **Траектория:** Они летят не строго вверх, а с небольшим разбросом (`random(-15deg, 15deg)`), замедляются и исчезают (`opacity: 0`).
* **Код:** CSS Animation `@keyframes floatUp`.



#### C. MANSUR'S PRESENCE (Эффект присутствия)

Бот не должен ощущаться как статичная картинка.

1. **Печатание (Typewriter Effect):**
* Сообщения в чате появляются по буквам. Скорость: 30ms на символ.
* Пока он "пишет", над его головой висит анимация `...` (Three dots bouncing).


2. **Слежение (Eye Tracking) - *Advanced*:**
* Если Таня кликает в левой части комнаты, спрайт Мансура зеркалится (flip) или поворачивает голову влево (если нарисуем такой спрайт). *Для MVP: просто легкий наклон.*



#### D. UI TRANSITIONS (Движение меню)

Никаких резких появлений (`display: none` -> `block`). Только плавные переходы.

* **Easing:** Используем `cubic-bezier(0.34, 1.56, 0.64, 1)` (эффект пружины/overshoot).
* **Bottom Sheet:** Меню магазина выезжает снизу вверх (`translateY(100%)` -> `translateY(0)`).
* **Toast Notifications:** Ачивки выплывают сверху и висят 3 секунды.

### 🔄 REVISED STEP 1.5.4: SELF-HOSTED DEPLOYMENT

Вместо облака мы используем классическую связку **Build -> Upload -> Nginx**.

#### 1. Сборка (На твоем компьютере)

Когда код будет готов, ты выполнишь одну команду, чтобы превратить React-код в набор статических файлов (HTML, CSS, JS):

```bash
npm run build

```

Это создаст папку `dist/`. В ней лежит вся твоя игра.

#### 2. Доставка на сервер

Ты просто копируешь папку `dist` на свой сервер (туда же, где живет бот, или в соседнюю папку).
*Пример (через SCP):*

```bash
scp -r dist/ user@your-server-ip:/var/www/tanya-clicker

```

#### 3. Настройка Web-сервера (Nginx)

Твой Python-бот работает на одном порту (например, слушает Telegram API), а Игру должен отдавать веб-сервер.
Лучший выбор — **Nginx**.

Я подготовлю для тебя **конфиг Nginx**, который ты просто вставишь на сервере.

**Пример конфига (`/etc/nginx/sites-available/tanya-game`):**

```nginx
server {
    server_name game.your-domain.com; # Твой домен

    root /var/www/tanya-clicker; # Куда ты положил папку dist
    index index.html;

    location / {
        try_files $uri $uri/ /index.html; # Важно для React роутинга
    }

    # SSL настройки (Certbot добавит их сам)
    listen 443 ssl; 
    # ... тут будут ключи сертификатов ...
}

```

#### 4. Связка с Ботом

В Python-коде ты укажешь свой домен:

```python
GAME_URL = "https://game.your-domain.com" 
# Или "https://your-server-ip/game", если настраиваешь через подпапки

```

## 12. 👁️ VISUAL OVERRIDE ENGINE (The "Cat Bed" Logic)

We encountered a logic problem: *How to hide the Cat sprite when the Cat Bed (which contains the cat) is purchased?*

### The Solution: Data-Driven Visibility

Instead of hardcoding `if (hasBed) hideCat()` inside React components, we moved the logic to the data layer.

1. **Data Layer (`items.json`):**
* Added field `visual_replaces: string[]`.
* Example for Bed: `"visual_replaces": ["item_cat_cream"]`.


2. **Render Logic (`Room.tsx` -> `useMemo`):**
* **Step 1:** Create a Set of `itemsToShow` based on inventory (`count > 0`).
* **Step 2:** Iterate through owned items. If an item has `visual_replaces`, add those target IDs to a `itemsToHide` Set.
* **Step 3:** Filter: `RenderList = itemsToShow - itemsToHide`.
* **Step 4 (Calibration Exception):** If `calibrationMode` is active, we bypass the hiding logic (or selectively show) so the developer can still adjust the hidden assets if needed.

## 13. 🚑 CRITICAL HOTFIXES

### A. The "Decimal Crash" (`wallet.kusi.gte is not a function`)

* **Issue:** `zustand/persist` rehydrated the `wallet` object from LocalStorage as a plain JS object (string/number), losing the prototype methods of `break_infinity.js` (`Decimal`).
* **Fix 1 (Reviver):** Added a custom JSON reviver to detect `{ mantissa, exponent }` shapes and restore them to `new Decimal()`.
* **Fix 2 (Hydration Guard):** Added `onRehydrateStorage` hook. It runs after load and force-wraps `state.wallet.kusi` in a helper function `safeDecimal()` which ensures the value is a valid `Decimal` instance before the app starts rendering.

### B. The "Ghost Asset"

* **Issue:** `item_cat_bed` was added to logic but lacked a sprite file, causing crashes or empty renders.
* **Fix:** Generated `item_cat_cream_in_cat_bed.png`, added to `assetMap.ts`, and updated `items.json` to link the ID `item_cat_bed` to this new sprite.
